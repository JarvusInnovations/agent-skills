# Build pipeline: esbuild → committed `.mjs`

The whole point of this pattern is that `npx skills add <owner>/<repo>` delivers a CLI that runs
immediately under plain `node`, with nothing to install. That requires compiling your TypeScript
source into one self-contained file and committing it. This doc covers the build, the shim, the
version stamp, the `--check` drift gate, and the off-PATH invocation problem.

Template: `references/templates/build-cli.ts`, `references/templates/invocation.ts`,
`references/templates/shim`, `references/templates/gitattributes`.

## 1. esbuild config

```ts
await esbuild.build({
  entryPoints: ["src/cli/bin.ts"],
  outfile: "skills/mytool/scripts/mytool.mjs",
  bundle: true,            // inline every dependency
  platform: "node",
  format: "esm",
  target: "node20",        // the floor we promise users
  legalComments: "none",   // strip vendored license blocks from the bundle
  define: { __MYTOOL_VERSION__: JSON.stringify(gitVersion()) },
  banner: {
    js: "#!/usr/bin/env node\n// GENERATED by scripts/build-cli.ts from src/cli/ — do not edit. Run `bun run build`.",
  },
});
```

Why these matter:

- **`bundle: true`** inlines `axi-sdk-js`, `@toon-format/toon`, and anything else, so the artifact
  has zero runtime deps. This is what kills the `npm install` step.
- **`format: "esm"` + `target: "node20"`** — modern Node, ESM. The banner's shebang makes the
  `.mjs` independently runnable; the "GENERATED … do not edit" line stops humans (and agents)
  editing the artifact instead of the source.
- **Pin your build deps to exact versions** (`axi-sdk-js`, `@toon-format/toon`, `esbuild`) in
  `package.json`. The committed bundle's bytes are an artifact; a floating `^` version makes the
  bundle diff on unrelated installs. (A verified example: a version-only bump produced a one-line
  bundle diff.)
- esbuild happily resolves `.js` import specifiers against `.ts` files, matching the
  `moduleResolution: "bundler"` convention — so write `import { main } from "./cli.js"` even
  though the file is `cli.ts`.

After a real (non-`--check`) build, `chmodSync(outfile, 0o755)` — esbuild writes 644 and the
bundle needs +x to be run directly.

## 2. The bash shim

Agents should invoke `scripts/mytool <command>`, not `node …/mytool.mjs`. A three-line shim makes
that work:

```bash
#!/usr/bin/env bash
# Shim so the bundled CLI can be invoked as `scripts/mytool <command>`.
exec node "$(dirname "$0")/mytool.mjs" "$@"
```

Commit it **executable** (`chmod 755 skills/mytool/scripts/mytool`). `Write` creates files 644, so
set the bit explicitly. Gotcha: never run the *shim* with `node` (`node …/scripts/mytool` →
"Invalid or unexpected token") — it's bash. Run the shim directly or run the `.mjs` with node.

## 3. Version stamping from `git describe`

Inject a real version so `mytool --version` is honest:

```ts
function gitVersion(): string {
  try {
    return execSync("git describe --tags --always", { encoding: "utf8" }).trim() || "dev";
  } catch {
    return "dev";
  }
}
```

Passed into esbuild via `define`. In `cli.ts`:

```ts
declare const __MYTOOL_VERSION__: string;
const VERSION = typeof __MYTOOL_VERSION__ === "string" ? __MYTOOL_VERSION__ : "dev";
```

### The release / version-stamp dance (hand-rolled — a candidate for an `axi build` helper)

Because the version comes from `git describe`, the committed bundle in a tagged commit and a fresh
rebuild of that tag are byte-identical only if you stamp *after* tagging:

1. merge the release,
2. tag the merge commit,
3. `bun run build` so `git describe` embeds the new tag,
4. commit the re-stamped bundle,
5. move the tag onto the stamped commit.

It's fiddly; flag it in your release docs.

## 4. The `--check` drift gate

CI must fail if someone edits `src/cli/` without rebuilding the committed bundle. Rebuild in
memory (`write: !check`) and compare to the committed file:

```ts
if (check) {
  const generated = result.outputFiles![0]!.text;
  const committed = readFileSync(OUTFILE, "utf8");
  if (normalizeVersion(generated) !== normalizeVersion(committed)) {
    console.error(`${OUTFILE} is out of date — run \`bun run build\` and commit the result`);
    process.exit(1);
  }
}
```

### Why `normalizeVersion` is essential

Committing the bundle changes the git sha, and CI checkouts may lack tags — so the embedded
`git describe` version *always* differs between the committed bundle and a fresh build. Without
normalization, `--check` would fail on every PR. Strip the version literal out of **both** sides
before comparing, so version drift never trips the gate while any real source change still does:

```ts
function normalizeVersion(source: string): string {
  // matches the compiled `typeof __MYTOOL_VERSION__ === "string" ? "v1.2.3" : "dev"` literal
  return source.replace(/"[^"]*" : "dev"/, '"__VERSION__" : "dev"');
}
```

Tune the regex to however your `VERSION` line compiles. Verify the gate works by changing an
*emitted string* (not a comment — `legalComments: "none"` strips comments, so a comment-only edit
correctly does **not** trip the gate).

## 5. Off-PATH invocation: `cliInvocation()`

The bundle ships inside a skill and is **not on `PATH`**. If any emitted output (home view, hook
output, `help[]` hints, error suggestions) prints a bare `mytool`, an agent assumes it's on PATH
and the call fails. Resolve the real invocation once and use it everywhere:

```ts
export function cliInvocation(): string {
  // prefer the sibling shim (…/scripts/mytool) when executable — matches how SKILL.md documents it
  // else fall back to `node <bundle>` which always works
  // collapse $HOME → ~ for readability
}
```

See `references/templates/invocation.ts` for the full implementation. Note: inside **SKILL.md**
itself, examples are hardcoded to the relative `scripts/mytool` form, because the skill's working
context is the skill directory — `cliInvocation()` is for runtime-emitted output.

**Add a prose backstop too.** `cliInvocation()` fixes the paths *you* emit, but an agent may still
copy a bare tool name it sees elsewhere. A one-liner in SKILL.md closes that gap cheaply (lavish-axi
does this): *"If output shows a follow-up command starting with `mytool`, run it as `scripts/mytool …`
— the CLI is not on PATH."* (An npm-distributed tool says `npx -y mytool …` instead.)

## 6. `.gitattributes`

The bundle is large (hundreds of KB, thousands of lines). Mark it generated so GitHub collapses it
in diffs and it doesn't dominate review:

```
skills/*/scripts/*-axi.mjs linguist-generated=true
# or for a single tool:
skills/mytool/scripts/mytool.mjs linguist-generated=true
```

## 7. Where the source lives (per layout)

- **Single-skill / app-monorepo:** `src/cli/`. One `OUTFILE` constant in `build-cli.ts`.
- **Multi-skill repo:** `packages/*/src/axi/`, with a `TARGETS: {entry, outfile}[]` array in
  `build-cli.ts` looping over each. Keep each package's own `tsconfig.json` excluding `src/axi/**`
  (esbuild bundles the CLI; tsc shouldn't emit it into `dist`), and add a root `tsconfig.axi.json`
  whose `include` lists every axi source dir for a dedicated `type-check:axi`.
